> 名詞對照見 [../中英名詞對照表.md](../中英名詞對照表.md)

---

## 0. 本科在本系的定位

在電子工程裡，數位設計的學習鏈大致是：

1. **數位邏輯設計（Digital Logic Design）**：布林代數、卡諾圖、組合邏輯、正反器、有限狀態機（FSM）。重點是「用閘（gate）想電路」。
2. **本科（HDL 與數位系統設計）**：學會用 **硬體描述語言（HDL, Hardware Description Language）** 把這些邏輯「描述」出來，再交給工具自動合成成閘級電路，並在 **FPGA** 或 **ASIC** 上實作。重點從「手畫電路」轉成「寫程式碼，但腦中想的是電路」。
3. **VLSI / 數位積體電路**：把同樣的 RTL 描述往下走到實體佈局（layout）、時脈樹、功耗、製程，是 ASIC 流程的後段。

所以本科是**數位邏輯與 VLSI/FPGA 實作之間的橋樑**：上承邏輯觀念，下接 EDA 工具流程。對通訊背景的擁有者來說，這也是把 OFDM、通道編碼、FFT 等演算法「落地成硬體」的入門關卡。

> **本科最重要、也最反直覺的一句話**：HDL 不是程式語言，是**電路描述**。你不是在「執行」程式碼，而是在「畫一張電路圖」。寫得像 C 一樣是初學者最常犯、後果最嚴重的錯誤。

## 1. 為什麼需要 HDL：軟體思維 vs 硬體思維

擁有者軟體很強，所以這一節最關鍵——**要刻意把軟體直覺關掉**。

| 觀念 | 軟體（C/Python） | 硬體（HDL/RTL） |
|---|---|---|
| 執行模型 | 指令**逐行循序**執行 | 所有電路**同時並行**存在、永遠在跑 |
| 變數 | 記憶體裡的一格 | 一條導線（`wire`）或一個暫存器（`reg`/正反器） |
| 賦值 | 把值存進去 | 建立一個**實體連接**或一個**邏輯閘** |
| 迴圈 | 重複執行 | 多半是**展開（unroll）成並排的硬體**，不是時間上重複 |
| 函式呼叫 | 跳轉、堆疊 | 通常是**複製一份硬體**（除非分時共用） |
| 「快」 | 演算法複雜度低 | 關鍵路徑延遲短、可拉高時脈、可平行 |

核心心法：**你寫的每一行可合成程式碼，最後都要對應到「實際存在的閘與正反器」。** 寫之前先問自己：「這描述出來的是組合邏輯（一坨閘）還是時序邏輯（正反器）？關鍵路徑會經過哪些東西？」如果你答不出來，那行程式碼八成有問題。

兩大主流語言：

- **Verilog**：語法近 C，簡潔、好上手，業界（尤其 IC 設計、北美、台灣）主流。其超集 **SystemVerilog** 加了強型別、`logic`、介面、驗證功能（class、constraint、assertion），現代設計與驗證幾乎都用它。
- **VHDL**：語法源自 Ada，**強型別、冗長、自我說明性高**，型別轉換嚴格（`signed`/`unsigned`/`integer` 間要顯式轉換）。歐洲、航太、國防偏好。

兩者表達能力相當，學會其一再讀另一個不難。本筆記以 **Verilog** 示範（程式碼片段僅示意硬體思維，語法細節從略）。

## 2. 抽象層次：行為、RTL、結構

同一個電路可以在不同抽象層描述，由高到低：

### 2.1 行為層（Behavioral）

只描述「做什麼」，不管硬體長相。常含 `initial`、延遲 `#10`、`for` 跑很多次、`$display` 等。**通常不可合成**，主要用在**測試平台**或建模參考。

```verilog
// 行為層：描述演算法，不是硬體
initial begin
  a = 0;
  #10 a = 5;        // 延遲 10 個時間單位，純模擬概念
  result = factorial(a);  // 函式像軟體一樣呼叫
end
```

### 2.2 暫存器傳輸層（RTL, Register Transfer Level）

**這是數位設計的主戰場。** 把電路想成「一堆暫存器（register）＋它們之間的組合邏輯」：每個時脈邊緣，資料從某些暫存器經過組合邏輯運算，被存進另一些暫存器。RTL 同時可模擬、又可合成，是設計的標準層次。

```verilog
// RTL：時脈邊緣把 (a+b) 存進 sum 暫存器
always @(posedge clk)
  sum <= a + b;     // 一個加法器 + 一排 D 正反器
```

### 2.3 結構層（Structural）

像畫電路圖一樣，**明確實例化（instantiate）子模組或基本閘並連線**。最接近實體網表（netlist）。

```verilog
// 結構層：明確接出一個半加器
xor (sum,  a, b);   // 內建閘原語
and (carry, a, b);
// 或實例化子模組：
adder u_add (.a(x), .b(y), .s(z));
```

實務上：**演算法/系統頂層用行為或高階 RTL，核心邏輯用 RTL，頂層整合用結構層把各模組連起來。** 三種混用很正常。

## 3. 可合成 RTL vs 不可合成構造

**合成（synthesis）** 是 EDA 工具把 RTL 翻成閘級網表的過程。並非所有 HDL 語法都能合成——能合成的東西必須對應到「真實硬體」。

### 3.1 可合成（會變成硬體）

- `assign`（連續賦值，組合邏輯導線）。
- `always @(*)` 描述組合邏輯；`always @(posedge clk)` 描述時序邏輯。
- `if/else`、`case`（變成多工器 mux／解碼邏輯）。
- 算術、邏輯、位移運算（變成加法器、比較器、移位器等）。
- 邊界固定的 `for`（被**展開**成並排硬體，不是時間迴圈）。
- 模組實例化、參數化（`parameter`、`generate`）。

### 3.2 不可合成（只能模擬）

- **時間延遲 `#N`**：沒有真實閘對應，合成時被忽略或報錯。
- **`initial` 區塊**：晶片沒有「程式開始時跑一次」的概念（FPGA 可用初值或 reset 取代）。
- `$display`、`$monitor`、`$finish`、檔案 I/O（`$readmemh` 算特例，可用於初始化記憶體）。
- `while`、不定邊界迴圈、`fork/join`、`wait`。
- 真實除法/取模（除非常數或 2 的冪），通常工具不直接合成。

**判準心法**：問「這在矽晶片上會是什麼閘或正反器？」答不出來，多半不可合成。`#N` 想表達「等一段時間」——但硬體的時間由時脈邊緣決定，不是你說等就等。

## 4. 阻塞 vs 非阻塞賦值（最重要的陷阱）

這是 Verilog 初學者**最常出錯、且模擬與合成行為會不一致**的地方。

- **阻塞賦值（blocking, `=`）**：在 `always` 區塊內**立即生效、循序執行**，像軟體一行行跑。下一行馬上看到新值。
- **非阻塞賦值（non-blocking, `<=`）**：在當前時間步**先全部計算右側、時間步結束才一起更新**。模擬出「所有正反器同時在時脈邊緣翻轉」的並行行為。

### 黃金準則（背起來，幾乎是業界鐵律）

1. **時序邏輯（clocked `always @(posedge clk)`）→ 用非阻塞 `<=`。**
2. **組合邏輯（`always @(*)`）→ 用阻塞 `=`。**
3. **同一個變數不要在同一 always 內混用 `=` 和 `<=`。**

### 為什麼？用兩段 shift register 體會

```verilog
// 想要：q1 <- d, q2 <- q1（兩級移位暫存器）
always @(posedge clk) begin
  q1 <= d;
  q2 <= q1;   // 非阻塞：q2 拿到的是「這次邊緣前」的舊 q1 → 正確，兩個正反器
end
```

若改成阻塞：

```verilog
always @(posedge clk) begin
  q1 = d;
  q2 = q1;   // 阻塞：q1 已先更新成 d，q2 直接拿到 d → 只剩一個正反器！
end
```

兩者**合成出來的硬體不同**。更糟的是混用時，模擬結果會依賴模擬器處理 `=`/`<=` 的順序（Verilog 標準未定義此順序），造成**競爭條件（race condition）**——模擬看似正常，上板卻錯，或不同工具結果不同。遵守黃金準則就能讓**模擬行為 = 合成硬體行為**，這是準則存在的根本理由。

## 5. 常見硬體陷阱

### 5.1 意外推斷出鎖存器（latch inference）

組合邏輯 `always @(*)` 裡，**只要有某條執行路徑沒給某訊號賦值，工具就認為「該訊號要保持原值」，於是合成出一個鎖存器（latch）。** 鎖存器是位準觸發、對毛刺敏感、難做靜態時序分析，FPGA 上幾乎都是 bug。

```verilog
// 危險：沒有 else，c=0 時 y 保持舊值 → 推斷 latch
always @(*)
  if (sel) y = a;
```

兩種根治法：

```verilog
// 法一：補完所有分支
always @(*)
  if (sel) y = a;
  else     y = b;

// 法二：開頭給預設值（最推薦，case 也適用）
always @(*) begin
  y = 1'b0;        // 預設值，保證每條路徑都有賦值
  if (sel) y = a;
end
```

`case` 同理：要嘛涵蓋所有情況、要嘛加 `default`，要嘛開頭給預設值。

### 5.2 敏感清單寫錯

舊式手寫 `always @(a or b)` 漏了訊號，會造成**模擬與合成不一致**（合成假設組合邏輯反應所有輸入，模擬卻不更新）。**一律用 `always @(*)`**（SystemVerilog 用 `always_comb`，並能在編譯期直接報出 latch），讓工具自動推敏感清單。

### 5.3 多重驅動 / 競爭

同一條 `wire` 被多個 `assign` 或多個 always 驅動，會衝突（除非是三態匯流排設計）。一個訊號只能有**一個驅動源**。

### 5.4 非同步 vs 同步重置、跨時脈域（CDC）

- 重置（reset）要決定同步或非同步，且全設計一致；非同步重置要做「非同步斷言、同步解除」避免恢復時的亞穩態。
- 訊號從一個時脈域進另一個時脈域（**跨時脈域 CDC**）必須加同步器（如雙正反器），否則會發生**亞穩態（metastability）**。這是真實晶片最常見的隱性 bug，模擬不一定看得到。

### 5.5 阻塞賦值放進時序邏輯、位寬不匹配

前者見第 4 節；後者（如 8-bit 接 4-bit）Verilog 會悄悄截斷或補零，不報錯，要自己留意。

## 6. 測試平台（Testbench）

設計寫完一定要驗證。**測試平台是一個只用於模擬、不可合成的頂層模組**，它：

1. 例化 **待測設計（DUT, Design Under Test）**。
2. 產生時脈、重置與輸入刺激（stimulus）。
3. 觀察輸出，**自動比對**是否符合預期（自我檢查 self-checking）。

```verilog
module tb;
  reg clk = 0, rst, d;
  wire q;

  dut u_dut (.clk(clk), .rst(rst), .d(d), .q(q));  // 例化 DUT

  always #5 clk = ~clk;     // 產生 100 MHz 時脈（不可合成，模擬專用）

  initial begin
    rst = 1; d = 0;
    #12 rst = 0;            // 解除重置
    #10 d = 1;
    #20 if (q !== 1'b1) $error("q 不如預期");  // 自我檢查
    #10 $finish;
  end

  initial $monitor("t=%0t d=%b q=%b", $time, d, q);
endmodule
```

要點：

- 測試平台**可以、也應該**大量用行為層構造（`initial`、`#`、`for`、`$display`、檔案讀入向量），因為它不上板。
- **自我檢查**勝過肉眼看波形：用已知正確的**參考模型（golden model）** 或預期值自動 assert，回歸測試才可靠。
- 進階的驗證方法學是 **UVM（Universal Verification Methodology）**，用 SystemVerilog 的物件導向與隨機約束（constrained random）建構可重用的驗證環境。
- `===`/`!==` 是含 `x`/`z` 的四值比較，測試平台常用來抓未初始化的 `x`。

## 7. 合成與時序約束

### 7.1 合成（Synthesis）

工具（Synopsys Design Compiler、Synplify、Vivado、Quartus 等）把 RTL 對應到目標技術的元件（FPGA 的 LUT/正反器/DSP/BRAM，或 ASIC 標準元件庫），並做邏輯最佳化。輸出是**閘級網表（netlist）**。

### 7.2 時序約束（Timing Constraints）與時序收斂

工具不知道你的電路要跑多快，**必須由你用約束告訴它**，否則時序分析無意義。業界標準語言是 **SDC（Synopsys Design Constraints）**，FPGA/ASIC 工具大多支援。

最核心的約束：

- **`create_clock`**：定義時脈週期與責任週期。**永遠最先寫**，因為其他約束都參照時脈。
- **`set_input_delay` / `set_output_delay`**：描述晶片邊界（I/O）相對外部時脈的時序預算，讓工具把片外延遲也納入。
- **時序例外**：`set_false_path`（此路徑不需時序檢查，如真正的非同步路徑）、`set_multicycle_path`（允許多個時脈週期完成，放寬要求）、`set_max_delay`。

### 7.3 setup / hold 與靜態時序分析（STA）

兩個正反器之間（由組合邏輯連接）的時序由兩個約束決定：

- **建立時間（setup）**：資料必須在時脈邊緣**之前**穩定一段時間。違反 → **慢路徑問題**，可靠的解法是**降低時脈頻率**或縮短關鍵路徑（拆級、加管線 pipeline）。
- **保持時間（hold）**：資料必須在時脈邊緣**之後**維持一段時間。違反 → **快路徑問題**，與頻率無關，靠加延遲（buffer）或工具修補修正。

工具用 **靜態時序分析（STA, Static Timing Analysis）** 不靠模擬、窮舉所有路徑檢查 setup/hold。**關鍵路徑（critical path）**——延遲最長的組合路徑——決定電路最高時脈。設計師讓設計達到目標頻率且無時序違規的過程叫 **時序收斂（timing closure）**。

> 通訊背景直覺：建立時間裕量像鏈路的時序預算（timing budget），關鍵路徑像最壞情況路徑。要加速就**插入管線暫存器**把長組合路徑切短——代價是延遲（latency）增加、面積變大，吞吐率（throughput）提升。

## 8. FPGA 實作流程

從 RTL 到一塊真正會動的 FPGA，標準流程：

1. **設計輸入（Design Entry）**：寫 RTL（Verilog/VHDL/SystemVerilog）。
2. **功能模擬（RTL / behavioral simulation）**：用測試平台驗證邏輯正確（還沒時序資訊）。
3. **合成（Synthesis）**：RTL → 閘級網表（映射到 LUT、FF、DSP、BRAM）。讀第一波合成警告——latch、隱性 truncation、未驅動訊號常在此暴露。
4. **實作（Implementation）**：
   - **映射（Mapping / Technology Mapping）**：把網表對應到該 FPGA 的實體資源。
   - **佈局與繞線（Place & Route, P&R）**：決定每個資源放哪、用哪些繞線資源連接。受時序約束驅動（timing-driven P&R）。
5. **時序分析（STA）**：用真實佈局後延遲檢查 setup/hold，看是否 timing closure。沒過就回頭改 RTL（加管線、改架構）或調約束。
6. **產生位元流（Bitstream Generation）**：把實作結果打包成 FPGA 設定檔。
7. **下載與板上驗證（Download / On-board）**：燒進 FPGA，用邏輯分析儀或 ILA（如 Vivado 的 Integrated Logic Analyzer）抓內部訊號實測。

關鍵體會：

- **FPGA vs ASIC**：FPGA 可重設定、開發快、單價高、頻率較低，適合原型與中小量；ASIC 一次性工程費（NRE）昂貴、週期長，但量大時單價低、效能功耗最佳。RTL 多半兩者通用，差別在後段流程與資源庫。
- **「模擬過 ≠ 上板會動」**：CDC 亞穩態、時序未收斂、reset 處理、I/O 約束錯誤，都是模擬看不到、上板才爆的問題。所以功能模擬、STA、板上驗證三者缺一不可。

## 9. 一條龍最小範例（同步計數器）

把前面觀念串起來——一個帶同步重置的 8-bit 計數器，這是標準可合成 RTL：

```verilog
module counter (
  input  wire       clk,
  input  wire       rst,     // 同步重置
  output reg  [7:0] count
);
  always @(posedge clk) begin   // 時序邏輯 → 非阻塞
    if (rst) count <= 8'd0;     // 有 else 等價（rst 為主），不會推 latch
    else     count <= count + 8'd1;
  end
endmodule
```

對照清單：時序邏輯用 `posedge clk` ＋ 非阻塞 `<=`；`if` 兩條路徑都賦值給 `count` 不推 latch；`count` 宣告為 `reg` 因為在 always 內賦值；位寬一致；加法器 + 8 個 D 正反器，關鍵路徑 = 加法器進位鏈，決定最高時脈。

## 10. 碩士延伸（簡短）

- **進階驗證方法學**：SystemVerilog Assertions（SVA）、UVM 驗證環境、約束隨機驗證（constrained random）、功能覆蓋率（functional coverage）。
- **形式驗證（Formal Verification）**：等價性檢查（RTL vs 網表）、模型檢查（model checking）以數學證明性質，而非靠模擬向量。
- **高階合成（HLS, High-Level Synthesis）**：從 C/C++/SystemC 自動生成 RTL，提升設計生產力（Vivado HLS、Catapult）。
- **低功耗設計**：時脈閘控（clock gating）、電源閘控、多電壓域、UPF/CPF 功耗意圖描述。
- **跨時脈域與時脈架構**：CDC 形式檢查、FIFO 同步、時脈樹合成（CTS）、多時脈/多模多角（MCMM）時序分析。
- **可測試性設計（DFT）**：掃描鏈（scan chain）、JTAG、內建自我測試（BIST）。
- **進階時序收斂**：物理感知合成、retiming、useful skew、管線與架構重構。

## 11. 博士研究方向（列表）

- 機器學習輔助 EDA：用 ML/RL 做佈局繞線、時序預測、合成參數調優。
- 大型語言模型（LLM）自動生成與驗證 RTL（含正確性保證、形式化檢核）。
- 高階合成的最佳化與可預測性（演算法到硬體的品質落差縮小）。
- 形式驗證的可擴展性：大型 SoC 的模型檢查、抽象與分解技術。
- 近似計算（approximate computing）在數位資料路徑的設計與驗證。
- 安全導向硬體設計：硬體木馬偵測、側通道防護的 RTL 級方法、資訊流追蹤。
- 容錯與可靠性：輻射環境（單事件翻轉 SEU）下的 RTL 強化與冗餘架構。
- 非同步（clockless）電路的設計方法學與工具鏈。
- 可重設定運算與 coarse-grained reconfigurable array（CGRA）架構與映射。
- 跨層次協同設計：演算法—架構—電路的自動探索（DSE, Design Space Exploration）。
