# E2 即時作業系統（Real-Time Operating Systems, RTOS）概論

> 嵌入式特化第 2 科。
> 名詞對照見 [中英名詞對照表 → 嵌入式](../中英名詞對照表-4.md#嵌入式)。

## 這科在學什麼

bare-metal 的「主迴圈 + 中斷」在功能變多時難以維護、難保證時序。RTOS 提供**多任務排程**與**同步原語**，讓多個工作各自像獨佔 CPU，並對時序提供可預測性。本科：

1. 即時性的定義與排程理論
2. 任務、排程器、上下文切換
3. 同步與通訊（號誌、互斥、佇列、事件）
4. 常見 RTOS（FreeRTOS、Zephyr、ThreadX 等）
5. 即時性陷阱與分析

## 第 1 章　什麼叫「即時」

即時 ≠ 快，而是**可預測 / 有截止期限（deadline）**：

- **硬即時（hard）**：錯過 deadline = 系統失敗（安全氣囊、馬達電流環、飛控）。
- **軟即時（soft）**：偶爾錯過只降低品質（影音串流、UI）。
- **韌即時（firm）**：錯過則該結果無用，但不致命。

關鍵指標：**最壞情況執行時間（WCET）**、**最壞反應時間**、**抖動（jitter）**。

## 第 2 章　為什麼需要 RTOS

bare-metal 的痛點：
- 大量狀態機糾纏、難擴充。
- 高優先工作可能被低優先長迴圈卡住。
- 阻塞式等待浪費 CPU。

RTOS 提供：
- 多任務並行的抽象（每任務各有堆疊與優先權）。
- 搶占式排程保證高優先即時反應。
- 阻塞時自動讓出 CPU（睡眠等事件）。

但代價：記憶體（每任務一份堆疊 + 核心開銷）、複雜度（競爭 / 死結 / 優先權反轉）。

## 第 3 章　任務與排程

### 3.1 任務狀態
Running、Ready、Blocked（等事件 / 延時）、Suspended。排程器在 Ready 集合中挑下一個跑。

### 3.2 上下文切換（Context Switch）
保存目前任務的暫存器 / 堆疊指標，載入下一個任務的——由排程器在 tick 中斷或主動讓出時觸發。切換有固定成本。

### 3.3 排程策略
- **固定優先權搶占式（最常見）**：永遠跑最高優先的 Ready 任務；同優先用時間片輪轉（round-robin）。
- **協作式**：任務主動讓出（少用，易被卡）。
- **EDF（最早截止優先）**：動態優先，理論利用率最高（可達 100%），但實作少見。

### 3.4 即時可排程性分析
- **速率單調（RMS / RM）**：週期越短優先越高。$n$ 個任務可排程充分條件：
$$\sum_i \dfrac{C_i}{T_i}\le n(2^{1/n}-1)$$
（$n\to\infty$ 約 0.69）。$C_i$ 執行時間、$T_i$ 週期。
- **EDF**：可排程條件 $\sum C_i/T_i\le 1$。
- 反應時間分析（RTA）更精確地考慮搶占與阻塞。

## 第 4 章　同步與通訊

多任務共享資源 → 需同步，否則資料競爭（race condition）。

### 4.1 臨界區與原子性
關中斷 / 排程器鎖 / 原子指令保護短臨界區。

### 4.2 互斥鎖（Mutex）
保護共享資源「一次一人」。RTOS 的 mutex 通常支援**優先權繼承**以緩解優先權反轉。

### 4.3 號誌（Semaphore）
- **二元號誌**：事件 / 中斷通知任務。
- **計數號誌**：管理一組相同資源。

### 4.4 佇列（Queue）/ 訊息
任務間傳資料的主力，天然解耦 + 同步（空則讀取者阻塞、滿則寫入者阻塞）。

### 4.5 事件旗標 / 通知
- 事件群組：等多個條件（AND/OR）。
- 任務通知（FreeRTOS）：輕量、快，取代部分號誌。

## 第 5 章　即時性的三大陷阱

### 5.1 優先權反轉（Priority Inversion）
低優先任務持有鎖，中優先任務插隊，導致高優先任務被間接卡住。
- 火星 Pathfinder 著名 bug。
- 解法：**優先權繼承**、**優先權天花板**。

### 5.2 死結（Deadlock）
多任務互相等待對方的鎖。預防：固定上鎖順序、逾時、避免巢狀鎖。

### 5.3 飢餓 / 活鎖
低優先任務永遠搶不到 CPU；或不斷重試卻無進展。

## 第 6 章　中斷與 RTOS 的關係

- 中斷優先權通常高於任何任務（硬體層）。
- ISR 內只用「FromISR」版 API（不可阻塞），把工作交棒給任務（**延後中斷處理 / 下半部**）。
- **中斷對任務的喚醒**：ISR 釋放號誌 → 喚醒等待的高優先任務 → 切換。

## 第 7 章　時間管理

- **系統 tick**：週期性中斷（如 1 ms）推動延時與時間片。
- **無滴答（tickless）模式**：閒置時關 tick 進低功耗，靠下一事件喚醒（省電關鍵）。
- 軟體計時器、延時 `vTaskDelay`、週期任務 `vTaskDelayUntil`。

## 第 8 章　記憶體管理

- 靜態配置（編譯期決定，最可預測，安全關鍵首選）vs 動態（heap_1~heap_5 等策略）。
- 每任務一份堆疊 → 堆疊大小估算與溢位偵測（watermark）。
- 記憶體保護單元（MPU）隔離任務（FreeRTOS-MPU、安全版）。

## 第 9 章　常見 RTOS 比較

| RTOS | 特點 |
|---|---|
| **FreeRTOS** | 最普及、輕量、開源（現屬 AWS）；生態 + 雲端整合（FreeRTOS-Plus）。 |
| **Zephyr** | Linux 基金會、模組化、內建驅動 / 網路 / 藍牙堆疊、device tree，現代且活躍。 |
| **ThreadX** | 工業級、認證齊全（Azure RTOS），微軟維護。 |
| **RT-Thread** | 中國生態廣、元件豐富。 |
| **µC/OS-III** | 教學經典、可認證。 |
| **VxWorks** | 高階商用、航太國防、強即時。 |
| **QNX** | 微核心、車用資安全（座艙、ADAS）。 |

選型考量：認證需求、記憶體足跡、生態 / 驅動、授權、即時性保證。

## 第 10 章　RTOS vs 通用 OS vs 裸機

- **裸機**：最省、最可控，但複雜功能難維護。
- **RTOS**：多任務 + 即時保證，KB 級足跡，無記憶體保護（多數）/ 無檔案系統（除非加元件）。
- **嵌入式 Linux（E4）**：完整 OS、MMU、檔案系統、網路，但延遲較大且不保證硬即時（需 PREEMPT_RT）。
- 混合架構：AMP（大核跑 Linux + 小核跑 RTOS），如 i.MX、STM32MP1。

## 第 11 章　驗證與除錯

- **追蹤工具**：Tracealyzer、SystemView——視覺化任務切換 / 中斷 / 阻塞。
- **WCET 分析**：靜態分析 + 量測。
- **壓力測試**：最壞負載下檢查 deadline。
- 常見 bug：堆疊溢位、優先權設定錯、ISR 用錯 API、競爭。
