# 韌體：連線模組整合

> 專題《IoT 聯網裝置》專題特有筆記（待產出清單第 3 份）。專題總覽見 [index](index.md)。
> 名詞對照見 [專題中英名詞對照表 → 連線模組整合](中英名詞對照表.md#連線模組整合)。

> 重心：MCU（host）如何**驅動外購通訊模組**。應用層協定（MQTT / TLS / TCP 語意）擁有者本就熟悉，本檔只留指標——火力集中在模組的硬體介面、AT 引擎與韌性設計這類嵌入式接合處。

## 第 1 章　模組是什麼、邊界在哪

外購模組是一顆別人做好的 SiP，內含**自己的 CPU + RF + 完整協定堆疊**（PHY/MAC、TCP/IP，常含 TLS）。對你而言它是個黑盒子，只暴露一個**主機介面（host interface）** 與幾支控制腳。

- 你（host）負責：透過介面下命令、收送 payload、管理模組電源與重連。
- 模組負責：上空中的一切（連 AP / 註冊基地台、建 TCP、TLS 加解密）。
- **這層抽象就是買模組的理由**：你不必自己實作 Wi-Fi/NB-IoT 協定與射頻。

## 第 2 章　主機介面（host interface）

| 介面 | 速度 | 場合 | 注意 |
| --- | --- | --- | --- |
| **UART + AT 指令** | 慢（數百 kbps） | 最常見、最好上手 | 大流量務必開硬體流控 RTS/CTS，否則溢位掉資料 |
| SPI（含自訂協定 / SLIP） | 中高 | 較高吞吐 | host 當 master，需處理模組就緒/中斷腳 |
| **SDIO** | 高 | Wi-Fi 高吞吐 | 接線與時序較嚴 |
| USB | 高 | 部分模組 | host 需 USB stack |

**控制腳**（別忘了接、別忘了在韌體控制）：
- `PWRKEY / ON` 開機、`RESET` 硬重啟、`EN/電源域控制`（睡眠時斷模組省電）。
- 狀態回授：`STATUS`、`RI`（振鈴 / URC 喚醒 host）、`DCD`、模組「網路就緒」腳。
- UART 流控：`RTS/CTS`。

## 第 3 章　AT 指令模型

- host 送 `AT+XXX=...\r\n`，模組回 `OK` / `ERROR` / `+CME ERROR: n`。
- **URC（主動上報）**：模組非同步丟出的通知，如「收到資料」「掉線」「來電」「註冊狀態變更」——**不對應任何命令**，隨時可能插進來。
- 兩種模式：
  - **命令模式**：一問一答。
  - **資料 / 透傳模式（transparent）**：進入後 UART 直接當資料管道；用跳脫序列（如 `+++`）切回命令模式。

## 第 4 章　寫一個 AT 引擎（核心工程）

這是整合的心臟。要素：

1. **命令佇列**：序列化送出，一次一條，配**逾時**與**重試**。
2. **回應解析狀態機**：`IDLE → SENDING → WAIT_RESP`，逐行緩衝，比對預期終結（`OK`/`ERROR`/特定前綴）。
3. **URC 分流**：收到的行先判斷是「目前命令的回應」還是「URC」；URC 丟給對應 handler（如資料到達 → 通知上層讀取）。
4. **資料/命令模式管理**：透傳切換、跳脫時序。
5. **非阻塞**：UART 收用中斷 + 環形緩衝（或 DMA），解析在主迴圈 / 任務做，**絕不在中斷裡 busy-wait 等 `OK`**。

開發訣竅：先用 USB-UART 接電腦，**用終端機手打 AT** 摸熟模組行為與時序，再寫程式；**log 每一筆 AT 往返**，問題九成看 log 就懂。模組的「AT command manual」是必讀聖經。

## 第 5 章　連線抽象：兩種架構

- **模組內 TCP/IP（socket-like AT）**：如 `...+CIPSTART` / `...+QIOPEN` 開 socket、送收資料。host 很輕，但功能受模組韌體限制。**IoT 最常用。**
- **PPP 撥號 + host TCP/IP stack（lwIP）**：模組只當數據機，host 自己跑 IP 堆疊。彈性最大，但 host 負擔重、RAM/ROM 需求高。
- TLS：通常**由模組 offload**（憑證預先寫入模組），host 只管 socket 與資料；需處理憑證更新與**時間來源**（憑證驗證要正確時間 → NTP / 模組時間 / RTC）。

> 之上的 MQTT / CoAP / HTTP、QoS、keepalive、TLS 握手原理：擁有者已熟，細節見 [E5 IoT 與邊緣運算](../../電機/09-嵌入式特化/E5-IoT與邊緣運算.md) 與 [19 計算機網路](../../電機/03-通訊特化/19-計算機網路.md)，此處不重述。

## 第 6 章　韌性設計（IoT 真正難的地方）

現場的模組會掉線、卡死、註冊失敗，韌體要能自己爬起來：
- **斷線重連狀態機**：偵測掉線（URC / keepalive 失敗）→ 退避重連（**指數退避 backoff** 避免狂連）→ 重建 socket → 恢復應用。
- **模組監督（supervisor）**：模組無回應 → 軟重啟（AT 重置）→ 還是死 → **硬重啟（拉 RESET / 斷電源域再上電）**。
- **看門狗**：硬體 WDT + 軟體 supervisor，避免整機卡死。
- **NB-IoT 註冊慢**：入網可能要數十秒甚至更久，狀態機要有耐心與逾時策略，不要誤判為失敗狂重啟。
- **緩衝與流控**：收發環形緩衝 + RTS/CTS，避免高速突發時溢位。

## 第 7 章　低功耗與模組

- **NB-IoT 的 PSM（省電模式）與 eDRX**：讓模組長時間深睡、只在約定窗口醒來收網路訊息，是 NB-IoT 能用電池跑數年的關鍵。
- host 配合：模組睡時 host 也睡，用 **RTC** 或模組的 `RI` 腳喚醒；甚至睡眠時直接**斷模組電源域**（醒來重註冊的代價 vs 省電的權衡）。
- 與 [硬體-電源與電池](硬體-電源與電池.md) 的峰值電流、儲能電容設計對接；與 [韌體-架構與電源管理](韌體-架構與電源管理.md) 的睡眠策略對接。

```
應用邏輯（取樣→打包→上傳→睡眠）
  └ 連線管理（重連狀態機、MQTT 會話）
      └ AT 引擎（命令佇列、回應/URC 解析）
          └ HAL：UART(+DMA/中斷) / GPIO 控制腳 / WDT
```
